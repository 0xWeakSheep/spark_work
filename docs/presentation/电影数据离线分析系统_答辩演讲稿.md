# 电影数据离线分析系统答辩演讲稿

> 适配文件：`电影数据离线分析系统_答辩PPT.pptx`  
> 建议总时长：8 到 10 分钟  
> 讲述方式：按 PPT 页码逐页讲解，可根据实际答辩时间删减。

## 第 1 页：封面

各位老师好，我们小组本次汇报的题目是《基于 Spark 的电影数据离线分析系统》。

这个项目围绕电影评分数据展开，目标是完成一个从数据预处理、数据存储、Spark 离线计算、结果入库，到 Web 页面可视化展示的完整分析流程。

系统主要用到了 Spark RDD、Spark SQL、Spark Streaming、MySQL 兼容数据库、Flask 和 ECharts。我们没有只做单个算法结果，而是把课程要求中的数据处理、平台搭建、计算分析和结果展示串成了一套可以运行和演示的系统。

## 第 2 页：考核目标拆解

这一页是我们对考核要求的拆解。

本项目主要完成了三个核心功能。

第一个功能是 RDD 离线分析。我们从 HDFS 输入目录读取电影和评分数据，对评分记录进行清洗和聚合，计算每部电影的平均评分，并输出平均评分最高的前 20 部电影。

第二个功能是 Spark SQL 加 MySQL 分析。我们把清洗后的电影、评分、用户数据导入数据库，再通过 Spark JDBC 读取结构化表，关联用户性别、评分记录和电影类型，统计男女用户对不同电影类型的关注情况。

第三个功能是 Streaming 动态趋势分析。我们把评分数据拆分成多个小批次文件，定时追加到流式输入目录，由 Spark Streaming 监听新文件，计算每个批次的评分数量、平均分、最高分和最低分，并写入结果表。

最后，所有分析结果通过 Flask API 提供给前端，页面使用 ECharts 展示图表。

## 第 3 页：总体架构

这一页展示的是系统整体架构。

系统从左到右分为数据层、计算层、存储层、服务层和展示层。

数据层主要是 MovieLens 电影数据，包括 `Movies.csv`、`Ratings.csv`、`Tags.csv` 和 `Users.dat`。数据经过脚本解压和清洗后，上传到 HDFS 兼容输入目录。

计算层由三类 Spark 作业组成：RDD 作业负责 Top 20 高分电影统计，Spark SQL 作业负责男女电影类型关注统计，Streaming 作业负责动态评分趋势统计。

存储层使用 MySQL 兼容数据库，既保存原始数据表，也保存分析结果表。这样 Web 页面不需要每次重新跑 Spark，而是直接查询结果表。

服务层由 Flask 提供 API，展示层由 ECharts 渲染图表。这样的设计把计算任务和页面展示解耦，演示时更加稳定。

## 第 4 页：数据集与预处理

这一页介绍数据集和预处理过程。

原始数据来自 `data/archive/moviedata-latest.rar`。解压后主要使用四类数据：电影表、评分表、标签表和用户表。

其中 `Movies.csv` 保存电影编号、标题和电影类型；`Ratings.csv` 保存用户评分记录；`Tags.csv` 保存用户打标签的数据；`Users.dat` 保存用户性别、年龄、职业和邮编信息。

预处理阶段主要做了三件事。

第一，解压 RAR 文件并整理目录结构。第二，使用 Python 的 `csv` 模块读取 CSV，避免电影标题中出现逗号时字段错位。第三，把 `Users.dat` 转换成标准 CSV，便于后续导入数据库。

最终清洗后的数据规模是：电影 45843 条，评分 1048575 条，标签 753155 条，用户 6040 条。

## 第 5 页：数据库设计

这一页是数据库设计。

数据库分为原始表和结果表两类。

原始表包括 `movies`、`ratings`、`users` 和 `tags`。这些表主要保存清洗后的源数据，方便 Spark SQL 通过 JDBC 读取。

结果表包括 `top_movies`、`gender_genre_attention`、`gender_top_genres` 和 `stream_rating_trend`。

其中 `top_movies` 保存 RDD 作业计算出的前 20 部高分电影；`gender_genre_attention` 保存不同性别对各电影类型的关注次数；`gender_top_genres` 保存男女各自关注最多的电影类型；`stream_rating_trend` 保存 Streaming 每个批次的统计结果。

这样设计的好处是，原始数据和分析结果分开保存，Web 页面只查询结果表，减少页面访问时的计算压力。

## 第 6 页：RDD 平均评分最高前 20 部电影

这一页介绍 RDD 分析功能。

RDD 作业读取 HDFS 兼容目录中的 `Movies.csv` 和 `Ratings.csv`。读取后先过滤表头和异常数据，再把评分记录转换成以 `movieId` 为 key 的键值对。

对于每条评分记录，我们将它转换为 `(movieId, (rating, 1))` 的形式。然后通过 `reduceByKey` 计算每部电影的评分总和和评分次数，再用评分总和除以评分次数得到平均分。

接下来，作业会把评分结果和电影信息进行 `join`，补充电影标题。最后按照平均分降序、评分次数降序、电影标题升序排序，取前 20 条写入 `top_movies` 表。

页面中的 Top 20 横向柱状图和下方表格都来自这张结果表。

## 第 7 页：Spark SQL 男女关注电影类型

这一页介绍 Spark SQL 分析功能。

这个功能的核心是统计不同性别用户对不同电影类型的关注数量。

首先，我们把 `movies`、`ratings`、`users` 三张表导入数据库。然后 Spark SQL 作业通过 JDBC 读取这三张表。

读取后，作业先用 `ratings` 关联 `users`，获得每条评分记录对应用户的性别；再关联 `movies`，获得电影的类型字段。

由于一部电影可能属于多个类型，例如 Action、Adventure、Drama 等，所以我们使用 `split` 和 `explode` 把 `genres` 字段拆成多行。然后按照 `gender` 和 `genre` 分组统计数量。

最终结果写入 `gender_genre_attention` 表，页面用分组柱状图展示男女对各类型电影的关注差异。

## 第 8 页：Streaming 动态评分趋势

这一页介绍 Streaming 动态评分趋势。

为了在离线数据基础上模拟实时数据，我们把 `Ratings.csv` 拆分成 8 个小批次文件，每个批次 1000 条评分记录。

演示脚本会定时把这些批次文件追加到流式输入目录。Spark Structured Streaming 使用文件流方式监听该目录，并设置每次处理一个新增文件。

每处理一个批次，系统会计算这个批次的评分数量、平均分、最低分和最高分。然后通过 `foreachBatch` 把结果写入 `stream_rating_trend` 表。

Web 页面每隔几秒轮询 `/api/stream-trend` 接口，因此折线图可以看到评分趋势随批次不断变化。

这个功能体现了 Spark Streaming 的基本处理流程：监听输入、按批处理、聚合统计、结果入库、页面刷新。

## 第 9 页：Flask + ECharts 可视化

这一页是 Web 可视化部分。

后端使用 Flask 提供四个主要接口。

第一个是 `/api/summary`，返回电影数量、评分数量、用户数量和类型数量。第二个是 `/api/top-movies`，返回评分最高前 20 部电影。第三个是 `/api/gender-genres`，返回男女电影类型关注统计。第四个是 `/api/stream-trend`，返回 Streaming 趋势数据。

前端使用 ECharts 展示三个核心图表：Top 20 高分电影柱状图、男女电影类型关注柱状图、实时评分趋势折线图。

这里我们把 ECharts 做成了本地静态文件，不依赖外部 CDN。这样在课堂演示时，即使网络不稳定，页面图表也可以正常加载。

## 第 10 页：运行命令与验收

这一页展示运行和验收方式。

项目使用 Docker Compose 统一管理运行环境。启动服务时执行：

```bash
docker compose up -d --build
```

初始化数据和批处理作业时执行：

```bash
docker compose exec spark-hadoop bash scripts/setup_all.sh
```

查看 HDFS 兼容目录时执行：

```bash
docker compose exec spark-hadoop hdfs dfs -ls /movie/input
```

启动 Streaming 演示时执行：

```bash
docker compose exec spark-hadoop bash scripts/streaming/run_streaming_demo.sh
```

验收结果包括：HDFS 输入目录能看到 4 个文件；数据库原始表有数据；`top_movies` 有 20 条结果；男女类型统计表有结果；Streaming 趋势表有 8 个批次；Web 页面可以显示三个核心图表。

## 第 12 页：总结

最后做一个总结。

本项目完成了一个基于 Spark 的电影数据离线分析系统。系统覆盖了数据解压清洗、HDFS 兼容存储、RDD 离线计算、Spark SQL 结构化分析、Streaming 动态趋势统计、数据库结果保存和 Flask + ECharts 可视化展示。

## 可选 Q&A 准备

### 问题 1：为什么用 Docker，而不是直接在 Mac 上安装 Hadoop 和 Spark？

因为 macOS 上直接安装 Hadoop、Spark、Java 和 MySQL 容易遇到版本兼容问题。Docker 可以把运行环境固定下来，便于重复部署和课堂演示，也不会污染本机环境。

### 问题 2：为什么性别统计不用 Tags.csv 作为核心数据？

因为 `Users.dat` 只有 6040 个用户，而 `Tags.csv` 的用户范围更大，两者直接关联会损失较多数据。评分表和用户表匹配更稳定，所以性别统计使用 `Ratings.csv + Movies.csv + Users.dat`，按电影类型作为统计口径。

### 问题 3：Web 页面为什么不直接调用 Spark？

Spark 作业启动成本比较高，不适合每次页面请求都执行。我们先让 Spark 把结果写入数据库，Web 只查询结果表，这样页面响应更快，系统分层也更清晰。

### 问题 4：Streaming 是真正实时吗？

本项目是课程演示环境下的准实时模拟。我们用分批追加文件的方式模拟评分数据持续到达，Spark Structured Streaming 监听输入目录并按批次处理。它体现了 Streaming 的处理思想，但不是生产级实时消息队列方案。

### 问题 5：系统还能怎么优化？

可以从三个方向优化。第一，增加更多分析维度，例如年份、类型、用户年龄段和职业。第二，引入 Kafka 作为实时数据源。第三，增加机器学习模块，例如评分预测和电影推荐。
